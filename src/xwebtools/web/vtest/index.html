IHE Virtual Testing Configuration (System = HIEOS)

Welcome! This page provides configuration details to support Internet based testing for HIEOS, Vangent's IHE conformant XDS.b/XCA implementation.

Contact information to support testing is provided at the end of this page.

Please note that we have currently enabled extensive logging on our system. This might result in some system latency.

TLS Configuration (optional)

The HIEOS platform uses 2-way TLS encryption. In our testing environment, the HIEOS server has been configured with a self-signed X.509 server
certificate. It has also been pre-configured to accept another self-signed X.509 certificate as a client certificate.

In order to enable your system to communicate with HIEOS using TLS, the following steps need to be followed:
  • Download and configure your truststore to accept our server certificate.
Additionally, perform one of the following two steps:
  • Download and configure your truststore and keystore with our test client certificate
    and corresponding private key.
  • Alternatively, provide us with your X.509 client certificate and we would configure the HIEOS server to accept it.

Patient Identifiers

Patient identifiers need to be registered with the XDS.b Registry before "Provide & Register" transactions can be issued. One of the following options
can be chosen to register patient ids with the XDS.b Registry:

  1. XDS.b Registry web services endpoints (Patient Identify Feed) have been exposed by HIEOS to accept HL7 V3 messages for
    patient add, update and merge transactions. Patient information can therefore be manipulated in HIEOS
    by invoking the mentioned services. Please ensure that patient identifiers are valid HL7 V3 representations.
    It is recommended that one of the assigning authorities from the table, XCA Key Parameters (near the end of this page),
    is used to qualify the patient id.

    XDS.b Registry "Patient Identity Feed" Web Service Endpoints
    Actor Transaction URL TLS? MTOM?
    Registry PRPA_IN201301UV02 (Add Patient) http://208.81.185.143:8080/axis2/services/xdsregistryb no no
    Registry PRPA_IN201301UV02 (Add Patient) https://208.81.185.143:8181/axis2/services/xdsregistryb yes no
    Registry PRPA_IN201302UV02 (Update Patient) http://208.81.185.143:8080/axis2/services/xdsregistryb no no
    Registry PRPA_IN201302UV02 (Update Patient) https://208.81.185.143:8181/axis2/services/xdsregistryb yes no

    Registry PRPA_IN201304UV02 (Merge Patient) https://208.81.185.143:8181/axis2/services/xdsregistryb yes no

  2. A valid patient identifier can be generated by going to Patient ID Generator.
    You can then use the generated patient identifier in subsequent XDS.b/XCA transactions that require a patient id.
  3. A valid patient identifier can be generated programmatically by using the rest service http://208.81.185.143:8080/xwebtools/pidallocate?rest=1.
    Similarly, you can then use the generated patient identifier in subsequent XDS.b/XCA transactions that require a patient id.
  4. You can use the preconfigured patient id "b1f770dad41c457^^^&1.3.6.1.4.1.21367.2009.1.2.315&ISO" which contains 3 valid
    documents (pdf, xml, txt) stored in our XDS.b Registry/Repository.

IHE XCA Test Configuration Details

Expected Behavior

XCA Expected Behavior
Actor Expected Behavior
Initiatiating Gateway
  • RegistryStoredQuery transactions that specify:
    • a patientId will be directed to the Initiating Gateway's local XDS.b Registry and also possibly IBM/NIST
      XCA Responding Gateways, depending upon the assigning authority for the patient id specified. One of the
      assigning authorities below must be present in the patient id.
    • this Initiating Gateway's homeCommunityId will be directed to the local XDS.b Registry.
    • the NIST homeCommunityId will be directed to NIST's XCA Responding Gateway.
  • RetrieveDocumentSet transactions that specify:
    • this Initiating Gateway's homeCommunityId will be directed to the local XDS.b Repository.
    • Otherwise, requests will be directed to the appropriate XCA Responding Gateway.
Responding Gateway
  • CrossGatewayQuery requests will be directed to the Responding Gateway's local XDS.b Registry.
  • CrossGatewayRetrieve requests will be directed to the Responding Gateway's local XDS.b Repository.
